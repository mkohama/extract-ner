# 判定ルール集（確信度・カテゴリ・マスク確定）

data-redactor が「何を・どのくらいの確からしさで機密と見なし・どう処理するか」を 1 本にまとめた
リファレンス。散在しがちな確信度（強・中・弱…）の決まり方とマスク確定の手順をここで引けるようにする。

- 対象読者：このツールの判定を理解・レビューする人。
- 概要は [README.md](README.md)、設計の経緯は `docs-dev/`（git 管理外）。本書が**現在の挙動の正**。
- 最終更新の根拠コミット：`feat(masking): … two-system consensus`（2系統合議＝2026-06-30 実装）。

---

## 0. 大前提（これだけは外さない）

- **recall 最優先**＝マスク漏れ＝情報漏洩。迷うものは伏せるか、レビューに回す。
- **「隠す」だけでデータは消さない**。微弱・除外も候補・データには残る（既定で非表示なだけ）。
- **ラベルは信じない**。種別（人名か社名か）は揺れる前提。芯は「**隠すべき物か否か**」の合議。
- 確定（自動マスクの最終的な拠り所）は**名簿（辞書）だけ**。検出器は誤るので確定にしない。

---

## 1. 全体像（入力 → 判定 → 取り扱い）

```
入力（ファイル/テキスト/kb-mcp）
  → テキスト化・チャンク分割
  → 各チャネルが「票」を出す
       ├ 辞書（dict）         … 名簿一致           → 確定の根拠
       ├ 正規表現（regex）     … 連絡先（メール等）   → 強の根拠
       ├ NER 系統             … Sudachi品詞 ∪ GiNZA(electra/ja_ginza)
       └ LLM 系統             … pii-masker（文脈判定）
  → 2系統合議（NER と LLM は対等な2系統）
       ├ ① カテゴリ（種別）を決める
       └ ② 確信度（強/中/弱…）を決める
  → ③ 取り扱い（自動マスク / 要レビュー / 非表示）をモードで振り分け
```

ポイントは **NER と LLM が対等な「2系統」**であること（§3）。

## 2. カテゴリ（6種）

| カテゴリ | 例 | マスク対象か | 確信度の決まり方 |
|---|---|---|---|
| **人名** | 田中、John Smith | ★特別（対象） | 系統合議（§4） |
| **社名** | オーロラ電子、Nebula | ★特別（対象） | 系統合議 |
| **商標** | 製品名・ブランド | ★特別（対象） | 系統合議 |
| **連絡先** | メール・電話 | 対象（主に regex） | regex 一致＝強／LLM 単独＝中（§3） |
| **地名** | 横浜市、長崎 | 対象外（弱＝要レビュー） | 必ず弱 |
| **その他** | 識別子・一般語 | 対象外 | 弱 → **微弱**（§9。LLM識別子だけ弱で免除） |

- **特別カテゴリ＝人名・社名・商標**。確信度の「強/中」を投票で決めるのはこの3つ。
- **連絡先の確定的な検出は正規表現（＝強）**（§3。メール等は形が決まっているため）。
  LLM も連絡先を出すが、単独なら中（regex が主役。regex が取りこぼした分を LLM が拾う位置づけ）。
- **地名・その他は票数によらず弱**（誤分類で人物が紛れるので必ず人がレビュー）。
- カテゴリは伏せ字の見た目（`[人物1]`/`[社1]` 等）を決めるだけ。確信度には効かない。

## 3. チャネルと「2系統」

票は `(チャネル, ラベル)` の形。チャネルは独立で、**走ったチャネルだけ**を集約する。

**確信度を数える2系統（対等）**

- **NER 系統**＝ `sudachi` ∪ `ja_ginza_electra` ∪ `ja_ginza`。
  - この3つは**「票」ではなく系統内の入力**。NER は内部で何チャネル一致しても、外に出す意見は **1つ**。
  - （フラットに数えると NER の3チャネルが LLM の1票を圧殺するため、等分にする。）
- **LLM 系統**＝ `llm`（pii-masker）。常に1系統。

**系統に数えない「決定的チャネル」**（それぞれ固有の確信度床を持つ）

| チャネル | 意味 | 確信度 |
|---|---|---|
| `dict` | 実辞書（名簿）一致 | **確定** |
| `session` | 昇格（他箇所で確認済みの語を再注入） | **強** |
| `regex` | 連絡先（メール等）の正規表現一致 | **強** |

- `session`（昇格）＝ある箇所で強になった clean な語を「確認済み」として全出現へ広げる仕組み。
  名簿ではないので**確定ではなく強**に留める。
- `regex` 連絡先は決定的だが誤検出があり得るので**確定でなく強**（除外リストで外せる）。

---

## 4. 確信度の決まり方 ★この章が核心

確信度は **NER×LLM の2系統合議**で決める。

| 確信度 | 条件 | 既定の扱い |
|---|---|---|
| **確定** | 辞書（名簿）一致 | 自動マスク |
| **強** | **2系統（NER∧LLM）がともに特別カテゴリを検出**（種別が違ってもよい）／ session ／ regex連絡先 | 自動マスク |
| **中** | **1系統のみ**が特別カテゴリを検出（floor：特別は必ず中以上＝弱に落とさない） | 要レビュー |
| **弱** | どの系統も特別を出さない（地名/その他のみ） | 要レビュー（自動マスク外） |
| **微弱** | 中/弱 のうち「コードらしき」表層（§9） | 既定で非表示・自動マスク外 |
| **除外** | allowlist で人が「機密でない」と外した語（§9） | 既定で非表示・自動マスク外 |

**読み方**

- 「強」は**2系統（NER∧LLM）がともに「これは隠すべき」と言った**＝強い証拠、という意味。
  - NER だけが何チャネル一致しても 1系統＝**中**まで。LLM が併走して初めて 2系統＝**強**。
  - ⚠ 前提は「**両系統の誤りが独立**」。コード識別子では ja_ginza と LLM が相関して誤爆し、
    ノイズなのに2系統一致＝強に化けることがある（§9・§10）。「強＝常に信頼」ではない。
- 「中」は片方の系統だけ＝**レビュー必須**。floor があるので特別カテゴリは弱に落ちない。
- LLM は1モデルなので **LLM 単独は構造上 最大「中」**（第2の意見が無い）。欠陥ではなく当然の帰結。

**当てはめ例**

| 候補 | NER系統 / LLM系統 | カテゴリ | 確信度 |
|---|---|---|---|
| Sony | 社名 / 社名 | 社名 | 強（2系統） |
| 種別割れ | 社名 / 人名 | 人名（重い方） | 強（2系統・種別違いでも可） |
| 小坂 | 人名（NER単独）/ 沈黙 | 人名 | 中（1系統） |
| 小浜 | 地名 / 人名 | 人名（特別が地名に勝つ） | 中（人名は LLM 1系統） |
| 長崎 | 地名 / 地名 or 沈黙 | 地名 | 弱（特別なし） |

> 小浜：旧フラット集計では「地名2票 vs 人名1票」で地名・弱に落ち、LLM の人名票が死票になっていた。
> 2系統合議では特別（人名）が地名に勝ち、人名・中に救済される。

**確信度は「出現ごと」。自動マスクの選択は「実体単位」（案2）**

- 確信度は**出現（span）ごと**に決める（上の合議は各出現で走る）。同じ表層でも、NER が当たった
  出現だけ 2系統＝強、他は LLM 単独＝中、と割れ得る。
- 自動マスクの初期選択は**実体単位**：ある表層に **確定/強 の出現が 1 つでもあれば、その表層の
  全出現を選ぶ**（`_auto_mask_spans`。CLI の `apply(expand=True)` と同じ）。
  - 例：`C社ファイル` が 4 出現、うち 1 出現だけ NER＋LLM で強・残り 3 は LLM 単独で中
    → 実体は社名・強 → **4 出現すべてを自動マスク**（1/4 しか隠れず 3 箇所漏れる、を防ぐ＝recall）。
- **同形異義語は壊れない**：`小浜`（`小浜市`=地名 / `小浜さん`=人名）は、どの出現も 2系統一致に
  ならず**中止まり**なので、この自動選択（強のみ）に**引っかからない**。出現ごとの区別は保たれる。
  - 一方の出現が真に強な同形異義語（例 人名 `小浜さん` が NER＋LLM で強）は、実体単位ゆえ他出現も
    初期選択に含まれるが、**出現ごとビューで個別に外せる**（同形異義語の個別制御はそのビューの役割・§7）。
- 系統単位で票を**和集合**にして実体の確信度を底上げする案（旧「案1」）は**採らない**：同形異義語で
  「NER=地名 と LLM=人名」を実体単位で合議すると地名の出現まで人名・強に化けるため（区別が壊れる）。

## 5. カテゴリ（種別）の決まり方

優先度：**人名 > 社名 > 商標 > 連絡先 > 地名 > その他**（`_CAT_PRIORITY`）。

### 5-0. 何をどのカテゴリとして扱うか（入力ラベル → カテゴリ）

各チャネルの生ラベルを 6 カテゴリへ写す表（`_NER_LABEL_CATEGORY` / `_ENE_TO_CATEGORY` /
`_sudachi_category`）。**ここで拾わないラベルは候補にすら上がらない**（＝票にならない）。

**NER（GiNZA electra / ja_ginza。関根の拡張固有表現ラベル）**

| カテゴリ | 採用するラベル | 備考 |
|---|---|---|
| 人名 | `PERSON` | `N_PERSON`（人数）は数値表現なので**含めない** |
| **社名** | `COMPANY` / `COMPANY_GROUP` | **この2つだけ**。下記の組織系（CORPORATION_OTHER 含む）は含めない |
| 地名 | `CITY` / `PROVINCE` / `COUNTRY` / `ADDRESS` / `POSTAL_ADDRESS` / `FACILITY_PART` / `STATION` / `AIRPORT` | 具体的な地名のみ。`_OTHER` 系は含めない |

- **社名から外した組織系ラベル**：`CORPORATION_OTHER`（企業その他）・`SHOW_ORGANIZATION`（興行団体）・
  `SPORTS_ORGANIZATION_OTHER`・`POLITICAL_ORGANIZATION_OTHER`・`POLITICAL_PARTY`・`GOVERNMENT`・
  `INTERNATIONAL_ORGANIZATION`。業務文書で一般語の誤爆（ゴミ）になりやすいため（`CORPORATION_OTHER`
  も粒度が粗く社名として怪しい）。**実在の組織は LLM(Company)＋辞書**で拾う。
- **地名から外した `_OTHER` 系**：`GPE_OTHER` / `GEOLOGICAL_REGION_OTHER` / `LOCATION_OTHER` /
  `DOMESTIC_REGION_OTHER` / `CONTINENTAL_REGION_OTHER` / `FACILITY_OTHER`。粒度が粗く誤爆しやすい。
- **連絡先は NER から取らない**（`@`・数字に過剰発火するため regex に委ねる）。`Product_Other` 系も
  ノイズ過多で除外。国籍・民族・一族（`Nationality`/`Ethnic_Group_Other`/`Family`）は会社でないので社名にしない。
- 補足：NER 単独の社名/人名票は元々**中**（自動マスク外）。上の絞り込みの主効果は**レビュー一覧の
  ノイズ削減**（recall への実害は小。地名はそもそも弱＝レビュー）。

**LLM（pii-masker の ENE type）**：`Person`→人名／`Company`・`Department`→社名／
`Province`・`City`・`Country`・`Address`・`Facility`→地名／`Email`・`Phone_Number`→連絡先／
`Trademark`→商標／`Employee_ID`・`Account`・`IP_Address`→その他（識別子）。

**Sudachi 品詞**：`固有名詞-人名`→人名／`固有名詞-地名`→地名／`固有名詞-一般`→その他
（未知の英字を社名に水増ししない）。

**決定的チャネル**：`dict`＝辞書のセクション（社名/商標/人名）／`session`＝昇格元のカテゴリ／
`regex`＝連絡先（§3）。

### 5-1. 合議でカテゴリを決める

1. **辞書一致があれば、そのカテゴリで確定**（最優先・名簿が砦）。
2. なければ各系統を**1カテゴリへ畳む**（系統内合議）。
   - 系統内でも「特別が地名/その他に勝つ・重い特別が勝つ」＝優先度最上位を採る。
3. 系統間でも同じ原理：**いずれかの系統が特別を出したらそれを採用**（地名/その他に埋もれさせない）。
   両系統が別の特別なら**重い方**。

**Stage 1：NER の特別票を surface で弱める（コードノイズ対策）**

ja_ginza 等が英字/コードを社名・人名へ誤爆するので、NER の特別票を surface で「その他」に落とす
（＝コードノイズが LLM と組んで「強」になる経路を断つ。`_system_category`）。カテゴリ非対称：

2軸で弾く（OR）。`_looks_like_code`＝記号/数字の「コード性」（カテゴリ非依存）。
`_looks_like_nonperson_latin`＝ラテン文字の「人名らしさの否定」（**人名専用**）。

- **人名**：`_looks_like_code` ／ **ラテン文字主体で Titlecase 語の並びでない**もの → その他。
  実在のラテン人名は Titlecase（`John`/`Smith`/`Anne-Marie`）なので、それ以外は識別子/術語の誤爆として明示的に弾く：
  - 全小文字（`theta`/`list`）／camelCase・PascalCase の途中コブ（`EndTime`/`MarkID`/`dElem`/`OutputItegrity`）
  - 全大文字（`FIARSL`）／2文字以上の全大文字語を含む複合（`Wafer ID` の `ID`）／記号・ブラケット混じり（`theta・「`）
  - **`John`/`Smith`/`Anne-Marie` は Titlecase ＝人名のまま残す**（雑に純ASCIIを一律降格しない）。
    ⚠ `McDonald`/`DeWitt` 等の途中大文字の外国姓は巻き込む（LLM 2系統＝強・辞書で救済される前提）。
  - かな/漢字を含む人名（`小坂`）は**信頼して残す**。`サブエラーコード` 等の**カタカナ汎用語**は表層で
    固有名と区別できず対象外＝辞書/除外リストの仕事。
- **社名**：`_looks_like_code`（記号・英数字・数字のみ・1文字）**のみ** → その他。
  `_looks_like_nonperson_latin` は**社名に当てない**（`IBM`/`SAP`/`Apple`/`eBay`/`PayPal` 等＝Titlecase
  でない正当な社名を守る）。
- LLM 票は弱めない（文脈判定を尊重）。

→ NER 単独のラテン人名ノイズ（`EndTime`/`theta・「` 等）は Stage 1 で「その他・弱」に収まり、一覧の
  「人名」から外れる。本物の Titlecase 人名（`John`）は残る。

## 6. 取り扱い（マスク方針）＝モードで分ける

①カテゴリ ②確信度 を受けて、どう処理するかをモードで決める。

| モード | 自動マスク | レビュー | 非表示 |
|---|---|---|---|
| **手動**（UI・人がレビュー。**現行**）| 確定・強 | 中・弱 | 微弱・除外 |
| **自動**（API・人なし。**将来・未実装**）| 中以上（確定・強・中）| ―（無人）| 弱・微弱・除外 |

- 現在は **手動モードのみ**稼働（`AUTO_MASK_CONFIDENCE = ("確定","強")` を既定チェック ON）。
- 自動モード（中以上を無人でマスク）は **API 化が前提＝未実装**（残件）。
- どちらでも **弱（地名/その他）は自動マスクしない**（utility 保持）。

## 7. 実体ごと と 出現ごと（確信度ラベルの読み方の注意）

UI には「実体ごと」「出現ごと」の2ビューがある。**マスクの最小単位は出現**（共有の選択集合 `mask_sel`）。
両ビューはこの 1 つの集合を読み書きするので、切替で選択は消えない。

- **実体ごと**の行に出る確信度は、**その語の全出現のうち最良（代表値）**。
- **自動マスクの初期選択は実体単位**（§4「案2」）：実体（表層）に 確定/強 の出現が 1 つでもあれば、
  その表層の**全出現**を初期選択する。
  - 例：`C社ファイル` が 4 出現、うち 1 が強・3 が中 → 実体は強 → **4 出現すべて自動選択**（「全 4/4」）。
  - 旧挙動（出現ごとに 確定/強 だけ選択）では「強なのに 1/4 しか選ばれず ⚠一部」になっていた＝漏れ。
    案2 でこの漏れは解消。
- **「⚠ 一部 n/全」**は、いまは**人が出現ごとビューで一部だけ選択/解除した**ときに出る
  （自動マスクの取りこぼしではない）。同形異義語（`小浜市` は隠す・`小浜さん` は残す 等）を
  出現単位で作り分けたケース。
- 実体チェックは**全出現が揃ったときだけ ON**（一部選択なら OFF＝⚠一部 表示）。

## 8. マスク確定の方法（手順）

1. 文書を読み込み、各タブ（NER 検出／LLM 検出）で検出を走らせる。
2. **マージ&確信度**タブで票を集約し、確信度づけ（§4）する。
3. 既定で **確定・強**にチェックが入る（自動マスク候補）。中・弱はレビュー候補として一覧表示。
4. レビューして、チェックを付け外し（実体ごと/出現ごと）→ **[✅ マスクを反映]**。
   - 「機密でない」と判断した語は **[🚫 選択を除外リストへ]** で allowlist に送る（§9・除外）。
5. 反映後の**マスク済みテキスト**と、**復元用の対応表（placeholder↔原語）**を得る。

- レビューの付け外し（手動の差分）は **`mask_draft` として content_hash 単位で永続化**。
  再起動・再解析・ビュー切替でも手動選択は消えない。
- **「確定」ステージ（finalized）は未実装**（残件）。現状は「下書き（mask_draft）＋マスク済みテキスト DL」まで。
  programmatic な mask/unmask API も未実装（将来）。

## 9. 微弱・除外（候補から消さず、既定で隠す2つ）

**微弱**＝「コードらしき」表層を中/弱から降格（社内コード/変数名/列挙子の誤ラベル対策）。

- 判定（`_looks_like_code` v2）：記号マーカー `_` `::` `@` `~` `[ ] { } = ! < > ; |` `\` を含む
  （`\` はファイルパス `C:\Group\ABC-Z\...` を NER が `Group\ABC-Z` と塊にして社名誤爆する対策。
  実在の人名・社名に `\` は無い）／
  ASCII のみ＋数字（`16D`/`1L`/`37D`）／数字・記号のみ（`7-410`）／漢字以外の1文字／
  **CJK区切り・括弧が ASCII英字に隣接**（`AP・`/`theta・「`/`会社「A」`＝スパン境界の人工物）。
  **隣接に限る**ので、区切りが仮名間の実在語（`C型補正値リミット・チェック結果`）や純カナの中黒名
  （`ジョン・スミス`/`ソニー・ミュージック`）は対象外＝守る／
  **先頭・末尾に垂れたハイフン類・中黒**（`AP-`/`R-`/`-C02`/`田中・`＝境界の人工物。内部ハイフンの
  `Coca-Cola`/`Anne-Marie` や長音 `ー` は対象外＝守る）／
  **不均衡な丸括弧**（open≠close。`LSMonito(`/`)Monitor`/`東芝（`＝NER スパンが括弧を巻き込んだ人工物。
  ブラケット `[ ] { }` は常にコードだが、丸括弧 `( )`（ASCII）・`（ ）`（全角）は `Sony (Japan)`/`会社（日本）`
  のような正当な社名に出るので**数の不均衡だけ**を artifact とみなす＝均衡した括弧つき社名は守る）。
- 加えて **全大文字ASCII（区切り込み。`WEXPC-YCD`）** も微弱へ落とす。ただし **社名・商標は除外**
  （`IBM`/`SAP` の略語社名・`MCSX` 等の全大文字な商標/型番を守る。人名の全大文字は Stage 1 で
  その他へ落ちる）。
- 加えて **「その他」カテゴリ全般（＝弱）** を微弱へ落とす（隠すべきか不明な低価値）。
  ＝Sudachi `固有名詞-一般` の「ただの固有名詞」（`装置A` 等）や、Stage 1 で特別→その他へ降格された
  ノイズ（`EndTime` 等）はここで既定非表示に。**地名（弱）は「その他」でないので据え置き**（要レビュー）。
- **チャネルは問わない（NER 専用ではない）**。中/弱 かつ コードらしき なら **NER・LLM どちらの検出でも**降格する。
  - ＝**識別子以外の LLM 検出（人名/社名/商標）が中・コードらしきなら、LLM 由来でも微弱に落ちる**。
- **唯一の免除＝LLM が識別子（Employee_ID/Account/IP_Address）と判定したもの**：中/弱でも降格せず弱で残す
  （`7-410` 型の社員番号が消えると recall 漏れ＝致命的なため）。
- **確定/強は降格しない**（チャネル問わず）。
  - コードらしき語が**強に化けないように Stage 1（§5）で NER 票を先に弱める**ので、記号/英数字コードは
    そもそも強に到達せず中→微弱に落ちる（`Em_OffOn::` も解消）。Stage 2 を強へ広げる必要は無い。
  - ⚠ 残る穴：**全大文字ASCIIを「社名」と判定**したもの（`EGPDPRY`/`FIARSL`）。社名は IBM 等を守るため
    弱めず、`_looks_like_code` でも拾えない → NER 社名 ＋ LLM Trademark で**強に残り得る**（§10・A4）。
- 既定で非表示・自動マスク外。データには残す（取りこぼさない）。

> ⚖ 設計上の論点（残件）：LLM が**文脈で**「これは人名」と判定したものまで微弱に落としてよいか（識別子以外）。
> 「**LLM が裏付けたものは守り、LLM が沈黙したものだけ落とす**」へ寄せるのが A4（LLM沈黙ゲート）。
> 現状は識別子以外の LLM 検出も落とす＝LLM の文脈判定を一部殺している。

**除外**＝allowlist で人が「機密でない」と外した語（最弱）。

- 文脈非依存・グローバル。既定は**完全一致**（正規化 surface）。既定で非表示・自動マスク外。
- **`embed: true`**（辞書 `embed` と対称）：登録語を**境界で内包**する候補を**丸ごと除外**する
  （例 `FB` embed → `GetFBData`/`FBData`/`FB_DATA`、`補正` embed → `用補正値`/`株式会社補正マーク`）。
  境界は**2 段の供給源**：**トークン間＝形態素境界**（SudachiPy。`補正`→`用/補正/値`＝日本語対応）と
  **トークン内＝`_split_identifier` のサブワード境界**（camelCase・区切り＝英 `GetFBData`）。どちらも
  単位の**途中は拾わない**（`FBI` の `FB`／`補正` の一部 `補` は不一致＝境界照合）。
  形態素境界を使うには `apply_allowlist` に候補を覆う `tokens` を渡す（`analyze`・`apply_allowlist_to_analysis`
  とも渡す）。辞書 embed は部分マスクだが、**除外は候補単位＝スパンを割らず丸ごと**落とす。
- **辞書（dict）票・連絡先（regex）は上書きしない**（確定アンカーは守る）。検出由来のみ外せる。
  ＝**確定 ＞ 除外**。embed 除外と辞書 embed マスクが同じ複合語に当たっても辞書が勝つ
  （例 辞書 `Smash`embed × 除外 `Type`embed の `SmashMarkType` → `Smash` は伏せられ Type 除外は効かない）。
- ⚠ **候補丸ごと除外の recall 注意**：embed 除外はスパンを割らず候補全体を外すので、実在社名が NER で
  1 スパンに塊化しその中に embed キーが当たると漏れる（`ABC` embed → 実在社名 `Group\ABC-Z` を丸ごと除外）。
  **社名/商標は辞書で守るのが基本**（辞書アンカーはクラスタ分割で先に切り出され確定になる＝除外が触れない）。
  embed は「機密名に出てこないと確信できる語」限定にする（案ii）。
- **完全一致 vs embed の使い分け**：embed はキーを含む**語族すべて**を外す（`オプション` embed →
  `ティーチングオプション` だけでなく今後の `○○オプション` 全部）。**汎用語**（`オプション`/`ステップ`/
  `本パッケージ` 等）は機密名に紛れ込む恐れがあるので、**その特定の語だけ外したいなら完全一致**で登録する。
  「この語は機密名に絶対出ない・語族ごと外してよい」と言い切れるときだけ embed にする。
